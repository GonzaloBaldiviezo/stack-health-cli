Repository metadata

What it checks: Presence of a .git directory.

Why it matters:

  • Traceability — Git history tells you who changed what, when, and why.
  • Collaboration — Without version control, branches and reviews disappear.
  • Automation foundation — CI/CD depends on git as the source of truth.
  • Debugging — Git blame and bisect are often your fastest path to root cause.
git init

Project manifest

What it checks: Presence of a dependency or build manifest such as package.json, pyproject.toml, go.mod, Cargo.toml, pom.xml, or build.gradle.

Why it matters:

  • Reproducibility — Exact dependencies and versions can be recreated.
  • Environment consistency — CI, production, and teammates install the same setup.
  • Audit trail — Dependency changes become reviewable and traceable.
  • Build clarity — The manifest defines how the project is run and built.
# Node.js
pnpm init

# Go
go mod init github.com/user/repo

Documentation

What it checks: Presence of a README.

Why it matters:

  • Onboarding — New contributors need a fast way to understand the project.
  • Maintenance — Good docs reduce tribal knowledge and single points of failure.
  • Credibility — A readable README signals care and maturity.
  • Usage clarity — People do not adopt tools they cannot understand quickly.
README checklist
- What the project does
- How to install it
- How to run it
- How to contribute

Automated checks

What it checks: A test script or the presence of test files such as *.test.ts, *.spec.ts, or __tests__.

Why it matters:

  • Reliability — Tests catch regressions before users do.
  • Confidence — Refactors become safer when behavior is covered.
  • Documentation — Tests act as executable examples of expected behavior.
  • Regression prevention — Fixed bugs stay fixed.
{
  "scripts": {
    "test": "vitest run"
  }
}

Code quality scripts

What it checks: Presence of lint or typecheck scripts in the project manifest.

Why it matters:

  • Consistency — Linters make code easier to read across a team.
  • Bug prevention — Many issues are caught before runtime.
  • Security — Static analysis can highlight risky patterns early.
  • Automation — CI can enforce standards before merge.
{
  "scripts": {
    "lint": "eslint src",
    "typecheck": "tsc --noEmit"
  }
}

CI readiness

What it checks: Presence of CI configuration such as .github/workflows, .gitlab-ci.yml, or azure-pipelines.yml.

Why it matters:

  • Automated enforcement — Quality checks run on every push.
  • Safety net — Broken code is blocked before it reaches main.
  • Visibility — Teams can see build and test status immediately.
  • Evidence — Deployments have a verifiable trail of what passed.
name: CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest

Next steps

If you are improving your project health, follow this order so each investment builds on the previous one.

  1. Start with repository metadata and project manifest.
  2. Add a README so other people can run and understand the project.
  3. Add automated tests.
  4. Add linting and type checking.
  5. Wire all of that into CI.
The CLI is intentionally simple right now: the score should be understandable, not magical.
